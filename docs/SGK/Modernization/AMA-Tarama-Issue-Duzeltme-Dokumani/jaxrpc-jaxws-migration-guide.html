BÖLÜM 01

Önce anla — neyi neye dönüştürüyoruz

JAX-RPC, 2000'lerin başında IBM ekosisteminin SOAP standardıydı. WSDL-first bir model; class'ları sen yazmazsın, WSDL'den üretilir. Konfigürasyon dosyaları (webservices.xml, *_mapping.xml, ibm-webservices-*.xmi) servletten ayrı bir dünyaydı. WAS Traditional bu dünyayı kendi içinde yönetiyordu.

JAX-WS bambaşka bir felsefe getirdi: annotation tabanlı, kod-merkezli. Bir XML dosyası okumana gerek yok. Class'ın üstüne @WebService yazıyorsun, container geri kalanı çözüyor. Liberty bu modeli native destekliyor; JAX-RPC'yi desteklemiyor. Bu yüzden dönüşüm zorunlu, opsiyonel değil.

Pratik fark

JAX-RPC: WSDL → tool → Java stub → XML config'le bağla. JAX-WS: Java class + @WebService → bitti. WSDL'i hâlâ kullanıyoruz ama container yönetiyor, sen değil.

Bu rehberin yapacağı şey

  • Eski uygulamayı standart Maven yapısına geçirmek
  • JAX-RPC'ye özgü tüm dosyaları (config, mapping, IBM extension) temizlemek
  • Service impl class'larına @WebService annotation eklemek
  • Bean class'larına JAXB annotation'larını yerleştirmek
  • pom.xml'i sıfırdan kurmak, Liberty üzerinde çalışır WAR üretmek
BÖLÜM 02

Tespit — Maven projesi mi, Eclipse projesi mi

İlk yapacağın şey: önündeki uygulama hangi tipte? Çünkü ilk adımlar buna göre değişiyor.

Maven projesi ise

pom.xml var, src/main/java var

Bölüm 03'ü atla, doğrudan Bölüm 04 (Keşif) ile başla.

Eclipse Dynamic Web Project ise

.project + WebContent klasörü var, pom.xml yok

Bölüm 03'ten devam et. Önce Maven yapısına çevirmen lazım.

Hızlı kontrol

terminal · proje kök dizininde
dir pom.xml
dir WebContent
dir .project

pom.xml görünüyorsa Maven. WebContent görünüyorsa Eclipse. İkisi de varsa karma bir durumun; muhtemelen yarım kalmış bir dönüşüm. O zaman Bölüm 03'ten yine geç, eksikleri tamamla.

BÖLÜM 03

Eclipse projesini Maven'e çevirme

Yedi adım. Her birini sırayla yap, atlama.

1 · Yedek al

windows · cmd
cd ..\
xcopy /E /I uygulamaAdi uygulamaAdi-backup
cd uygulamaAdi

Git kullanıyorsan ek olarak commit at: git add -A && git commit -m "migration oncesi". Hata yaptığında geri dönecek bir yer her zaman olsun.

2 · Maven klasörlerini oluştur

windows · cmd
mkdir src\main\java
mkdir src\main\resources
mkdir src\main\webapp\WEB-INF
mkdir src\test\java

3 · Java kodlarını taşı

Eclipse projelerinde Java kodu genelde src\ doğrudan altında olur (örn: src\com\firma\uygulama\). Bunu yeni yere taşı:

windows · cmd
:: var olan paketi yeni yere taşı
xcopy /E /I src\com src\main\java\com

:: doğrula
dir src\main\java\com

:: doğruysa eski klasörü sil
rmdir /s /q src\com

İçindeki paket adı com değilse (tr, org, vb.) o adı kullan.

4 · WebContent içeriğini taşı

windows · cmd
xcopy /E /I WebContent\META-INF src\main\webapp\META-INF
xcopy /E /I WebContent\WEB-INF  src\main\webapp\WEB-INF
Önemli

WSDL dosyaların büyük ihtimalle WebContent\WEB-INF\wsdl\ içinde. Bu taşıma ile birlikte hedef yer olan src\main\webapp\WEB-INF\wsdl\'ye geliyorlar — doğru yer orası, taşıma sonrası başka yere kaydırma. Sebep Bölüm 05'te.

5 · Properties dosyalarını resources'a taşı

Eski projelerde properties dosyaları (app.properties, i18n*.properties) genelde src\ kökünde veya WebContent\WEB-INF\classes\ içinde durur. Hangisinden olduğunu bul, hepsini src\main\resources\ içine taşı.

6 · Eski klasörleri kaldır

windows · cmd
rmdir /s /q WebContent
del .classpath
del .project
rmdir /s /q .settings

7 · pom.xml oluştur

Boş bir pom.xml dosyası aç. İçeriğini Bölüm 09'da bulacaksın. Hemen yazma — önce keşif yap (Bölüm 04), çünkü pom'un içine girecek dependency listesi onun çıktısı.

kenar notu Burada en çok hata, "WebContent'i silmek" adımında oluyor. Silmeden önce taşımanın gerçekten başarılı olduğunu gözle doğrula. Bir keresinde xcopy'nin sessizce başarısız olduğu bir senaryoyla karşılaştım — taşıdığımı sandığım WSDL dosyası yeni yerde yoktu, silmiştim, geri dönüş yok. Yedek olmasa bitmiştim.

BÖLÜM 05

WSDL'i yerinde bırak

Eski projelerde WSDL src/main/webapp/WEB-INF/wsdl/ içinde olur. Bunu aynen orada bırakıyoruz. Başka yere taşıma — özellikle META-INF/wsdl'e taşıma.

Neden burada

Liberty, WEB-INF/wsdl/ içindeki WSDL'i otomatik tanıyor. Annotation'da wsdlLocation = "WEB-INF/wsdl/ServisAdi.wsdl" yazınca bu yolu direkt bulup yüklüyor. Bu, en az direnç gösteren yol — ve sahada her seferinde çalışan tek yol.

META-INF/wsdl tuzağı

Internet'te "WSDL'i META-INF/wsdl'e koy" diyen bir sürü tutorial var. Denedim, sahada çalışmadı. Liberty bazen WSDL'i kendisi üretiyor (annotation-only mod), bazen senin WSDL'inle senin WSDL'in arasında karışıyor; sonuç olarak SOAP yanıtlarında parametre isimleri arg0, arg1 olarak geliyor. İstemci tarafı patlıyor. Bu yola sapmak için iyi bir sebep yoksa sapma.

Sonuç olarak dosya yapın şu olmalı

hedef yapı
src/
└── main/
    ├── java/
    │   └── com/firma/uygulama/...
    ├── resources/
    │   └── *.properties
    └── webapp/
        ├── META-INF/
        │   └── MANIFEST.MF
        └── WEB-INF/
            ├── web.xml
            └── wsdl/
                ├── ServisAdi.wsdl
                └── *.xsd

"Şüpheli — önce içine bak" kategorisi

  • Servlet class'ları: extends HttpServlet yapan ama doGet/doPost içinde gerçek iş yapan bir sınıf varsa tutuyoruz. Sadece JAX-RPC için var olan (ör. servletten servisi açan generic class) varsa siliyoruz.
  • Filter class'ları: Custom logic varsa tut. JAX-RPC'ye özgüyse sil.
  • Log4jInit.java tarzı dosyalar: Bunlar genelde silinir. Bölüm 14'te detay var.
BÖLÜM 07

Tutulacak dosyalar

Aşağıdakiler kalıyor — ama bazılarına dokunmamız gerekecek.

  • Service impl class'lar — üzerlerine @WebService ekleyeceğiz (Bölüm 12).
  • Bean / DTO class'lar — JAXB annotation alacaklar (Bölüm 13).
  • WSDL ve XSD dosyaları — WEB-INF/wsdl/ içinde, dokunma.
  • DAO, Manager, Util, Helper sınıfları — custom kodun, olduğu gibi kalır.
  • Properties dosyaları — src/main/resources/'a taşınmış olarak.
  • Tek bir log4j2.xml — diğer log4j varyantları (prod, test, component.properties) silinir.
BÖLÜM 08

web.xml'i minimal hale getir

Eski web.xml içinde JAX-RPC servlet tanımları ve listener'lar olabilir. Hepsi gider. Liberty annotation'ları kendi okuyor; servisi web.xml'de tanımlamana gerek yok.

Silinecek bölümler

  • <servlet> blokları (eğer sadece WS için varsa)
  • <servlet-mapping> (yine sadece WS için olanlar)
  • <listener> — Log4jInit, IBM listener'ları
  • <context-param> — log4j config path gibi şeyler

Minimal web.xml

src/main/webapp/WEB-INF/web.xml
<?xml version="1.0" encoding="UTF-8"?>
<web-app xmlns="https://jakarta.ee/xml/ns/jakartaee"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="https://jakarta.ee/xml/ns/jakartaee
         https://jakarta.ee/xml/ns/jakartaee/web-app_5_0.xsd"
         version="5.0">

    <display-name>UygulamaAdi</display-name>

</web-app>

İçeride uygulamaya özgü ekstra ihtiyaç (security-constraint, filter, vb.) varsa onları koru. Sadece JAX-RPC artığı olanları temizle.

Neden jaxws-maven-plugin yok

Bazı tutorial'larda görürsün — class'ları WSDL'den üretmek için. Bizim ihtiyacımız yok. Mevcut Java class'larımız var ve onları kullanıyoruz; WSDL sadece referans olarak WEB-INF/wsdl/'de duruyor. Plugin'i koyarsan generated-sources üretir, kafa karıştırır. Koyma.

Neden webResources bölümü yok

WSDL src/main/webapp/WEB-INF/wsdl/'de, yani Maven'in varsayılan web kaynak yolunda. webResources ile başka yerden kopyalamana gerek yok. Eklersen tehlikeli — kopya WSDL'ler oluşur, hangi WSDL'in çalıştığını anlayamazsın.

BÖLÜM 11

Custom JAR'lar — Maven Central'da olmayanlar

Bazı JAR'lar şirket içi olur; Central'da bulamazsın. Üç seçenek var, sırasıyla tercih.

Tercih 1

Şirket Nexus / Artifactory

Şirketin bir repository sunucusu varsa pom.xml'e <repositories> ekle, oradan çek. En temiz yol.

Tercih 2

Local install

mvn install:install-file ile lokal Maven cache'ine kur. Tek geliştirici için pratik, takım için sürdürülemez.

Tercih 3

System scope

JAR'ı projeye lib/ klasörüne koy, pom'da <scope>system</scope> ile path göster. En son çare.

Tercih 2 — komut

terminal
mvn install:install-file ^
    -Dfile="./lib/custom-lib-1.2.jar" ^
    -DgroupId=com.firma ^
    -DartifactId=custom-lib ^
    -Dversion=1.2 ^
    -Dpackaging=jar

Tercih 3 — pom satırı

pom.xml içinde
<dependency>
    <groupId>com.firma</groupId>
    <artifactId>custom-lib</artifactId>
    <version>1.2</version>
    <scope>system</scope>
    <systemPath>${project.basedir}/lib/custom-lib-1.2.jar</systemPath>
</dependency>
System scope ek adım

System scope JAR'lar varsayılan olarak WAR'a girmez. maven-war-plugin'in packagingIncludes'ına eklemen lazım, yoksa Liberty NoClassDefFoundError fırlatır.

BÖLÜM 12

@WebService — sade ve doğru kullanım

Service implementation class'a @WebService ekliyoruz. Yalın hali kullanılır — çoğu örnekteki ek attribute'lar bizim için sorun çıkarır.

com/firma/uygulama/servis/ServisAdiImpl.java
package com.firma.uygulama.servis;

import jakarta.jws.WebService;

@WebService(
    serviceName = "ServisAdi",
    portName    = "ServisAdiPort",
    targetNamespace = "http://uygulama.firma.com/",
    wsdlLocation = "WEB-INF/wsdl/ServisAdi.wsdl"
)
public class ServisAdiImpl {

    public Sonuc islemYap(Musteri musteri) {
        // iş mantığı
        return new Sonuc();
    }
}

Bu yaklaşımın dört kuralı

  1. endpointInterface KULLANMA. Bir SEI (Service Endpoint Interface) varsa ve onu işaret edersen Liberty SEI'yi okuyup gerisini görmezden geliyor; impl'deki method değişikliklerin uygulanmıyor. Annotation'ı doğrudan impl'e yaz, SEI'ye değil.
  2. @WebMethod ve @WebParam EKLEME. Bunlar mevcut method imzasını kendi WSDL'inle eşleştirmek için gibi görünüyor — gerçekte WSDL üretimini değiştiriyor, parametre isimlerin arg0'a düşüyor. wsdlLocation ile dış WSDL'i bağladığımız için gerek yok.
  3. wsdlLocation kullan. Bu sayede istemciler eski WSDL'i görmeye devam ediyor — backward compatibility kırılmıyor.
  4. serviceName ve portName WSDL'de yazan isimlerle birebir aynı olmalı. Tek harf fark — Liberty servisi tanımıyor, 404 dönüyor.
benim ders aldığım yer @WebMethod(operationName="...") eklemenin daha "doğru" olduğunu sanıp ekledim. Soap mesajları yine geliyordu ama içindeki parametre adı arg0 oldu. İstemci eski parametre adıyla gönderdiği için unmarshalling sessizce null'la sonuçlandı. Bu noktayı bulmak yarım günümü aldı. Sade tut, fazlasını yapma — bu işin altın kuralı.
BÖLÜM 13

@XmlAccessorType(FIELD) — bean class'larına

Servisin alıp döndürdüğü her bean / DTO class'a JAXB annotation'ı eklemen lazım. Tek satır:

com/firma/uygulama/model/Musteri.java
package com.firma.uygulama.model;

import jakarta.xml.bind.annotation.XmlAccessType;
import jakarta.xml.bind.annotation.XmlAccessorType;

@XmlAccessorType(XmlAccessType.FIELD)
public class Musteri {

    private String ad;
    private String soyad;
    private int yas;

    // getter / setter'lar
}
Neden FIELD, neden PROPERTY değil

Varsayılan değer PUBLIC_MEMBER — JAXB hem field'ları hem getter/setter'ları okumaya çalışıyor, aynı alan iki kez serialize ediliyor: bir <ad>Veli</ad>, bir <Ad>Veli</Ad>. FIELD ile sadece field okunuyor, getter/setter sadece kodun çağırması için kalıyor. Temiz XML üretiyor.

Bu annotation'ı her bean class'a ekle. Atlanan tek bir bean varsa o class'ın geçtiği SOAP mesajı patlar.

BÖLÜM 14

log4j temizliği

Eski projelerde sıklıkla şu dosyalar bulunur:

  • log4j2.xml
  • log4j2-prod.xml
  • log4j2-test.xml
  • log4j2.component.properties
  • Log4jInit.java
  • visolog3.json veya benzeri uygulamaya-özgü log config'leri

Yapacaklar

  1. Tek bir log4j2.xml tut. Diğer environment-specific olanları (-prod, -test) sil. Environment'a göre log seviyesi Liberty'de jvm.options üzerinden yönetilir, dosya seçimiyle değil.
  2. Log4jInit.java'yı sil. Eski WAS'ta servlet üzerinden log4j'yi başlatma pattern'iydi. log4j2 classpath'ten log4j2.xml'i kendi buluyor; init class'a ihtiyaç yok.
  3. log4j2.component.properties ve visolog3.json gibi dosyaları sil — eğer uygulamanın gerçek bir bileşeni onları kullanmıyorsa. Kullanıp kullanmadığına emin değilsen grep / findstr ile referansını ara: findstr /s /i "visolog3" src\main\java. Çıktı boşsa sil.

Minimal log4j2.xml

src/main/resources/log4j2.xml
<?xml version="1.0" encoding="UTF-8"?>
<Configuration status="WARN">
    <Appenders>
        <Console name="Console" target="SYSTEM_OUT">
            <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss.SSS} %-5level %logger{36} - %msg%n"/>
        </Console>
    </Appenders>
    <Loggers>
        <Logger name="com.firma.uygulama" level="info"/>
        <Root level="warn">
            <AppenderRef ref="Console"/>
        </Root>
    </Loggers>
</Configuration>
BÖLÜM 15

Liberty server.xml

WAR'ı Liberty'de çalıştırmak için server.xml'in iki temel bölümü olmalı: feature listesi ve data source (uygulama DB kullanıyorsa).

server.xml
<server description="uygulama-adi">

    <featureManager>
        <feature>jakartaee-10.0</feature>
        <feature>xmlWS-4.0</feature>
        <feature>xmlBinding-4.0</feature>
    </featureManager>

    <httpEndpoint id="defaultHttpEndpoint"
                  host="*"
                  httpPort="9080"
                  httpsPort="9443"/>

    <application id="uygulama-adi"
                 location="uygulama-adi.war"
                 name="uygulama-adi"/>

    <!-- DB gerekiyorsa -->
    <library id="oracleJdbcLib">
        <fileset dir="${shared.resource.dir}/jdbc" includes="ojdbc8.jar"/>
    </library>

    <dataSource id="appDS" jndiName="jdbc/appDS">
        <jdbcDriver libraryRef="oracleJdbcLib"/>
        <properties.oracle
            URL="jdbc:oracle:thin:@host:1521/SERVICE"
            user="kullanici"
            password="sifre"/>
    </dataSource>

</server>

Liberty 23+ versiyonlar için: jaxws-2.2 yerine xmlWS-4.0; jaxb-2.2 yerine xmlBinding-4.0. Eski feature isimleri çalışmaz.

BÖLÜM 18

Sık görülen hatalar

Hata 01

SOAP yanıtında arg0, arg1

Sebep: WSDL Liberty tarafından üretiliyor, dış WSDL bağlanmamış. Çözüm: wsdlLocation'ı kontrol et, WSDL WEB-INF/wsdl/'de mi? @WebMethod / @WebParam annotation'larını kaldır.

Hata 02

NoClassDefFoundError

Sebep: pom'da bir dependency eksik, ya da scope yanlış (provided olmaması gereken provided olmuş). Çözüm: WAR içindeki WEB-INF/lib'i unzip ile bak — eksik JAR neyse pom'a ekle.

Hata 03

XML çıktıda field iki kez

Sebep: Bean class'a @XmlAccessorType(FIELD) eklenmemiş, varsayılan PUBLIC_MEMBER hem field hem getter okuyor. Çözüm: Her bean'e annotation'ı ekle.

Hata 04

Application failed to start

Sebep: webservices.xml veya ibm-webservices-*.xmi hâlâ WAR içinde. Çözüm: WAR'ı unzip ile aç, varlarsa source'tan sil ve yeniden build et.

Hata 05

404 — service not found

Sebep: serviceName / portName WSDL'deki ile eşleşmiyor, ya da context root farklı. Çözüm: WSDL'deki service name="..." ve port name="..."'u birebir kopyala.

Hata 06

java.lang.LinkageError

Sebep: pom'a jakarta.servlet-api, jakarta.jws-api gibi Liberty'nin zaten sağladığı API JAR'ları girmiş. Çözüm: Pom'daki Jakarta API dependency'lerine scope=provided ekle, ya da hepten sil (zaten jakarta.jakartaee-api içeriyor).
